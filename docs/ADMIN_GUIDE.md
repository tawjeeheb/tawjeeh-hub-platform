# دليل المالك — لوحة الإدارة (Owner / Admin Panel)

لوحة `/admin` هي مركز إدارة المتجر: المنتجات، الطلبات، العملاء، وسجل الأحداث.
هذا الدليل يشرح ما تفعله، وما الذي يعمل الآن، وما الذي يحتاج مفاتيح إنتاجية.

> **مهم — الوصول للوحة يتطلب Supabase.** تسجيل الدخول والتحقق من دور «المالك»
> يتمّان عبر Supabase Auth. بدون مفاتيح Supabase لا يمكن لأحد تسجيل الدخول، لذا
> تبقى `/admin` غير قابلة للوصول (تعيد التوجيه إلى `/auth/login`). هذا **حماية
> مقصودة**، وليس نقصًا — لا توجد «لوحة ديمو» تتجاوز المصادقة.

## إنشاء أول مالك (admin)

1. شغّل `supabase/schema.sql` ثم `supabase/seed.sql` على مشروع Supabase.
2. عبّئ مفاتيح Supabase الثلاثة (انظر `docs/SUPABASE_SETUP.md`).
3. سجّل حسابًا عاديًا من `/auth/signup`.
4. رقِّ الحساب إلى مالك عبر service role فقط (لا يمكن للعميل ترقية نفسه):
   ```bash
   node --env-file=.env.local scripts/bootstrap-admin.mjs you@example.com
   ```
5. سجّل الدخول — ستظهر لك `/admin`.

## إدارة المنتجات — `/admin/products`

- **جدول كامل:** العنوان، الـ slug، المجال، السعر، الحالة، حالة الملف، آخر تحديث.
- **فلترة** حسب الحالة + **بحث** بالعنوان/الـ slug/المجال.
- **حالة الملف** تُعرض كـ «مرتبط / غير مرتبط» — **لا يُعرض المسار الخام أبدًا**.
- **إضافة / تعديل** عبر نموذج مُتحقَّق بـ Zod.
- **أرشفة / استعادة** بنقرة (إجراء حالة آمن).
- **حذف** نهائي.

### حالات المنتج

| الحالة | عام؟ | شراء؟ | في الـ sitemap؟ |
| --- | --- | --- | --- |
| `available` (متاح) | نعم | نعم | نعم |
| `coming_soon` (قريبًا) | نعم | لا | نعم |
| `draft` (مسودة) | **لا** | لا | **لا** |
| `archived` (مؤرشف) | **لا** | لا | **لا** |

قواعد مفروضة في الخادم (`src/lib/validations.ts` + `src/lib/data/products.ts`):

- المنتج `available` يحتاج **سعرًا > 0** ووصفًا **≥ 40 حرفًا**.
- `draft`/`archived` لا يظهران في المتجر ولا خريطة الموقع ولا صفحة المنتج.
- لا يمكن نشر منتج ناقص حتى عبر زر تغيير الحالة السريع.
- `file_path` يُحفظ خادميًا فقط ولا يُعرض للعملاء إطلاقًا.

## إدارة الطلبات — `/admin/orders`

- عرض: رقم الطلب، العميل، الملف، المبلغ، الحالة، التاريخ. مع فلترة بالحالة.
- **للعرض فقط.** لا يوجد زر «تحويل إلى مدفوع» يدويًا.
- **لماذا؟** التفعيل (الـ entitlement الذي يسمح بالتحميل) يتم **حصريًا** عبر
  webhook دفع موقّع وموثّق (`/api/webhooks/payment`). منح «مدفوع» يدويًا يعني منح
  وصول دون دفع فعلي مؤكَّد — لذا هو **ممنوع بحكم التصميم**.

## نظرة عامة — `/admin`

- بطاقات: المنتجات، الطلبات، المدفوعة، العملاء.
- تفصيل المنتجات والطلبات حسب الحالة.
- **تنبيهات تشغيلية:** Supabase غير مهيأ، منتجات بلا ملف، الدفع تجريبي،
  `NEXT_PUBLIC_SITE_URL` على localhost.
- روابط سريعة + مراجع التشغيل (المتطلبات المؤجّلة، checklist النشر).

## ما الذي يعمل الآن مقابل ما يحتاج مفاتيح

| القدرة | بدون مفاتيح | يحتاج |
| --- | --- | --- |
| تصفّح المتجر العام + قواعد الإخفاء (draft/archived) | ✅ يعمل (من seed) | — |
| تسجيل الدخول للوحة الإدارة | ❌ | Supabase Auth |
| عرض/إضافة/تعديل/أرشفة/حذف المنتجات | ❌ (يحتاج دخول) | Supabase + service role |
| عرض الطلبات والعملاء | ❌ (يحتاج دخول) | Supabase |
| تحميل ملف PDF | ❌ | Supabase Storage + ربط `file_path` |
| دفع حقيقي + تفعيل الطلب | ❌ | مزوّد دفع + `PAYMENT_WEBHOOK_SECRET` |

> ترقية قاعدة بيانات قائمة لإضافة `draft`/`archived`:
> شغّل `supabase/migrations/0001_product_status_draft_archived.sql`.

التفاصيل الكاملة للمؤجّلات في `docs/DEFERRED_PRODUCTION_REQUIREMENTS.md`.
